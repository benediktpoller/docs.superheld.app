Die Superheld API unterstützt zwei Authentifizierungsmechanismen: **API-Schlüssel** für direkte Integrationen und **OAuth2** für delegierten Zugriff durch Drittanwendungen. Beide Verfahren verwenden scope-basierte Autorisierung.

## API-Schlüssel

### Format

API-Schlüssel verwenden folgende Präfixe:

| Präfix | Umgebung |
|---|---|
| `sh_live_` | Produktionsumgebung |
| `sh_test_` | Testumgebung |

Beispiel:

```
sh_live_abc123xyz456
```

### Erstellung

API-Schlüssel werden im Superheld-Dashboard unter **Einstellungen → API-Zugang → Neuer Schlüssel** erstellt. Bei der Erstellung wählen Sie einen Namen, die gewünschten Scopes und optional ein Ablaufdatum. Der Schlüssel wird nur einmal vollständig angezeigt — speichern Sie ihn sofort in einem Credential-Manager.

### Verwendung

API-Schlüssel werden im `Authorization`-Header übergeben:

```bash
curl -X GET https://api.superheld.app/api/v1/devices \
  -H "Authorization: Bearer sh_live_abc123xyz456"
```

Schlüssel dürfen nicht im Query-String oder Request-Body übermittelt werden. Alle Anfragen müssen über HTTPS erfolgen.

### Rotation

Empfohlene Vorgehensweise zur Schlüsselrotation:

1. Neuen Schlüssel mit identischen Scopes im Dashboard erstellen.
2. Integration aktualisieren — neuen Schlüssel in der Anwendung hinterlegen.
3. Alten Schlüssel widerrufen, sobald die Integration mit dem neuen Schlüssel arbeitet.

Empfohlener Rotationszyklus: alle 90 Tage. In sicherheitskritischen Umgebungen: 30 Tage.

### Widerruf

Kompromittierte Schlüssel können unter **Einstellungen → API-Zugang** sofort widerrufen werden. Der Widerruf ist unwiderruflich und wirkt sofort — alle Anfragen mit diesem Schlüssel erhalten ab diesem Zeitpunkt `401 Unauthorized`.

Für Szenarien, in denen Drittanwendungen im Namen eines Benutzers auf die API zugreifen.

### Authorization Code Flow

Für Webanwendungen, bei denen ein Benutzer den Zugriff explizit autorisiert:

```bash
# 1. Benutzer zur Autorisierung weiterleiten
https://auth.superheld.app/authorize?
  response_type=code&
  client_id=CLIENT_ID&
  redirect_uri=https://meine-app.de/callback&
  scope=devices:read threats:read

# 2. Authorization Code gegen Token tauschen
curl -X POST https://auth.superheld.app/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=authorization_code&code=AUTH_CODE&client_id=CLIENT_ID&client_secret=CLIENT_SECRET&redirect_uri=https://meine-app.de/callback"
```

Die Antwort enthält ein `access_token` und ein `refresh_token`. Das Access-Token ist 60 Minuten gültig.

### Client Credentials Flow

Für Server-zu-Server-Integrationen ohne Benutzerinteraktion:

```bash
curl -X POST https://auth.superheld.app/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=client_credentials&client_id=CLIENT_ID&client_secret=CLIENT_SECRET&scope=threats:read scan:execute"
```

```json
{
  "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "threats:read scan:execute"
}
```

### Token verwenden

OAuth2-Token werden wie API-Schlüssel im `Authorization`-Header übergeben:

```bash
curl -X GET https://api.superheld.app/api/v1/threats \
  -H "Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9..."
```

## Scope-basierte Autorisierung

Scopes begrenzen den Zugriff eines Schlüssels oder Tokens auf bestimmte Ressourcen und Aktionen. Die folgende Tabelle enthält alle Scopes, die für die kanonischen Endpunkte (siehe [API-Übersicht](/experts/api/)) erforderlich sind.

| Scope | Berechtigung | Status |
|---|---|---|
| `devices:read` | Geräte auflisten und Details abrufen | Documented |
| `devices:write` | Geräte registrieren und konfigurieren | Documented |
| `threats:read` | Bedrohungen auflisten und Details abrufen | Documented |
| `alerts:read` | Benachrichtigungsregeln anzeigen | Documented |
| `alerts:write` | Benachrichtigungsregeln erstellen/ändern | Documented |
| `scan:execute` | Inhalte zur Analyse übergeben | Documented |
| `status:read` | Schutzstatus abrufen | TODO: Scope mit Engineering bestätigen |
| `events:read` | Ereignisse abrufen (SIEM-Integration) | TODO: Scope mit Engineering bestätigen |
| `rules:write` | Schutzregeln erstellen und ändern | TODO: Scope mit Engineering bestätigen |
| `family:read` | Familienprofile abrufen | TODO: Scope mit Engineering bestätigen |
| `family:write` | Familienprofile erstellen und ändern | TODO: Scope mit Engineering bestätigen |
| `whitelist:write` | Whitelist-Einträge hinzufügen | TODO: Scope mit Engineering bestätigen |
| `reports:read` | Bedrohungsberichte abrufen und generieren | TODO: Scope mit Engineering bestätigen |

Anfragen, deren Scope nicht ausreicht, werden mit `403 Forbidden` beantwortet:

```json
{
  "error": {
    "code": "insufficient_scope",
    "message": "Der Scope 'threats:read' ist für diesen Endpunkt erforderlich.",
    "status": 403
  }
}
```

Vergeben Sie immer nur die minimal notwendigen Scopes. Eine Integration, die ausschließlich Bedrohungen überwacht, benötigt lediglich `threats:read`.

:::caution
**Sicherheitshinweise für API-Schlüssel:**
- API-Schlüssel niemals in Quellcode oder Versionskontrolle speichern — nutzen Sie Umgebungsvariablen oder Credential-Manager (1Password, HashiCorp Vault etc.)
- Separate Schlüssel für Entwicklung, Test und Produktion verwenden
- Bei langlaufenden Operationen (> 60 min) bei OAuth2 Refresh-Tokens implementieren, um Unterbrechungen zu vermeiden
:::
