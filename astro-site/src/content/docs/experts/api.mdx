Die Superheld API ermöglicht die programmatische Integration von Bedrohungserkennung, Geräteverwaltung und Alerting in bestehende Sicherheits-Workflows. Diese Seite beschreibt die zentralen Designprinzipien, das Authentifizierungsmodell und typische Integrationsmuster.

## API-Prinzipien

Die API folgt dem **REST**-Architekturstil. Alle Endpunkte akzeptieren und liefern **JSON**. Die Basis-URL lautet:

```
https://api.superheld.app/api/v1/
```

Jede Anfrage muss über **HTTPS** erfolgen — unverschlüsselte Verbindungen werden abgelehnt. Die API ist **versioniert**: Das Präfix `/v1/` garantiert Abwärtskompatibilität innerhalb einer Hauptversion. Änderungen an bestehenden Feldern oder Endpunkten werden erst mit `/v2/` eingeführt.

**Rate Limiting** schützt die Infrastruktur. Pro API-Schlüssel sind **120 Anfragen pro Minute** erlaubt. Bei Überschreitung antwortet die API mit `429 Too Many Requests`. Die Header `X-RateLimit-Limit`, `X-RateLimit-Remaining` und `X-RateLimit-Reset` informieren über den aktuellen Status.

## Authentifizierungsmodell

Alle Anfragen erfordern einen gültigen API-Schlüssel im `Authorization`-Header:

```
Authorization: Bearer sh_live_abc123xyz456
```

API-Schlüssel werden im Dashboard unter **Einstellungen → API-Zugang** erstellt. Für erweiterte Szenarien unterstützt die API zusätzlich **OAuth2** mit dem Authorization-Code-Flow und Client-Credentials-Flow. OAuth2-Token erhalten **Scopes**, die den Zugriff granular steuern — beispielsweise `devices:read`, `threats:read`, `alerts:write` oder `scan:execute`. Details zur Authentifizierung finden Sie auf der Seite [Authentifizierung und Autorisierung](/experts/auth/).

## Multi-Tenant-Modell

Die API arbeitet mit **organisationsbasierter Mandantentrennung**. Jeder API-Schlüssel ist genau einer Organisation zugeordnet. Anfragen liefern ausschließlich Daten der zugehörigen Organisation — ein übergreifender Zugriff ist nicht möglich. Geräte, Bedrohungen und Alerts sind strikt isoliert. Organisationsadministratoren können mehrere API-Schlüssel mit unterschiedlichen Scopes für verschiedene Integrationen erstellen.

## Zentrale Endpunkte

Die wichtigsten Ressourcen der API:

| Methode | Endpunkt         | Beschreibung                              |
|---------|------------------|-------------------------------------------|
| `GET`   | `/devices`       | Alle registrierten Geräte abrufen         |
| `GET`   | `/threats`       | Erkannte Bedrohungen auflisten            |
| `GET`   | `/alerts`        | Konfigurierte Benachrichtigungsregeln     |
| `POST`  | `/scan`          | Inhalt zur Bedrohungsanalyse übergeben    |

### Beispiel: GET /threats

```json
{
  "data": [
    {
      "id": "thr_8f3a2b",
      "type": "phishing",
      "severity": "high",
      "source": "email",
      "detected_at": "2026-03-14T09:22:17Z",
      "description": "Gefälschte Login-Seite erkannt"
    }
  ],
  "has_more": true,
  "total_count": 42
}
```

### Beispiel: POST /scan

```json
{
  "scan_id": "scn_4e7d1c",
  "status": "completed",
  "threat_detected": true,
  "threat_type": "phishing",
  "confidence": 0.97,
  "details": "Domain imitiert bekannten Finanzdienstleister. SSL-Zertifikat seit 2 Tagen aktiv."
}
```

## Event-Modell

Bedrohungen durchlaufen in der API einen definierten Lebenszyklus: **Erkennung → Klassifizierung → Benachrichtigung → Behebung**. Bei der Erkennung wird ein Threat-Objekt mit Typ, Schweregrad und Quelle erstellt. Die Klassifizierung ordnet die Bedrohung einer Kategorie zu (`phishing`, `malware`, `social_engineering`). Konfigurierte Alert-Regeln prüfen den Schweregrad und lösen Benachrichtigungen über den gewählten Kanal aus. Über die API können Bedrohungen anschließend als behoben markiert werden.

## Integrationsmuster

### Pull-Integration (Polling)

Bei der Pull-Integration fragt Ihr System die API regelmäßig nach neuen Daten ab. Das eignet sich für Dashboards, Berichte und periodische Synchronisierungen. Beachten Sie das Rate Limit von 120 Anfragen pro Minute — ein Polling-Intervall von 30 Sekunden pro Endpunkt ist empfehlenswert.

```bash
# Alle 30 Sekunden neue Bedrohungen abfragen
curl -X GET "https://api.superheld.app/api/v1/threats?since=2026-03-14T09:00:00Z" \
  -H "Authorization: Bearer sh_live_abc123xyz456"
```

### Push-Integration (Webhooks)

Bei der Push-Integration sendet Superheld Ereignisse aktiv an Ihren Endpunkt. Sie konfigurieren eine Alert-Regel mit `channel: "webhook"` und einer Ziel-URL. Bei jeder relevanten Bedrohung sendet die API einen `POST`-Request mit dem vollständigen Event-Objekt an Ihre URL. Webhooks sind effizienter als Polling und ermöglichen Echtzeit-Reaktionen.

```bash
curl -X POST https://api.superheld.app/api/v1/alerts \
  -H "Authorization: Bearer sh_live_abc123xyz456" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Kritische Bedrohungen",
    "severity_filter": ["high", "critical"],
    "channel": "webhook",
    "webhook_url": "https://mein-server.de/hooks/superheld"
  }'
```

**Empfehlung:** Nutzen Sie Webhooks für zeitkritische Sicherheitsereignisse und Polling für regelmäßige Bestandsabgleiche. Kombinieren Sie beide Muster für eine robuste Integration, bei der Webhooks die primäre Quelle bilden und Polling als Fallback dient.
