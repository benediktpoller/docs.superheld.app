Die Superheld API ermöglicht die programmatische Integration von Bedrohungserkennung, Geräteverwaltung und Alerting. Diese Seite beschreibt Endpunkte, Authentifizierung, Paginierung und Fehlerformate.

## Basis-URL und Versionierung

Alle Endpunkte verwenden die Basis-URL:

```
https://api.superheld.app/api/v1/
```

- Transport: ausschließlich HTTPS. Unverschlüsselte Verbindungen werden abgelehnt.
- Versionierung: Das Präfix `/v1/` kennzeichnet die aktuelle API-Version. `TODO-ENG-037`: Abwärtskompatibilitätsgarantien und Deprecation-Mechanismus bestätigen.
- Datenformat: JSON (Request und Response).
- Kanonischer Vertrag: [OpenAPI 3.1 Spezifikation](/openapi.yaml) (Draft — Engineering-Review ausstehend).

## Authentifizierung

Alle Anfragen erfordern einen gültigen API-Schlüssel oder ein OAuth2-Token im `Authorization`-Header:

```
Authorization: Bearer sh_live_abc123xyz456
```

Details zu API-Schlüsseln, OAuth2-Flows und Scopes: [Authentifizierung und Autorisierung](/experts/auth/).

## Rate Limiting

`TODO-ENG-043`: Rate-Limits mit Engineering bestätigen.

Pro API-Schlüssel sind derzeit **120 Anfragen pro Minute** dokumentiert. Bei Überschreitung antwortet die API mit `429 Too Many Requests`. Relevante Response-Header:

| Header | Beschreibung |
|---|---|
| `X-RateLimit-Limit` | Maximale Anfragen pro Zeitfenster |
| `X-RateLimit-Remaining` | Verbleibende Anfragen im aktuellen Zeitfenster |
| `X-RateLimit-Reset` | Unix-Timestamp, wann das Zeitfenster zurückgesetzt wird |

## Paginierung

Die API verwendet cursor-basierte Paginierung. Antworten mit mehreren Ergebnissen enthalten ein `has_more`-Feld und einen `cursor`-Wert für die nächste Seite:


## Fehlerformat

`TODO-ENG-038`: Fehlerformat mit Engineering bestätigen.

Die API liefert Fehler in folgendem Format:

```json
{
  "error": {
    "code": "error_code",
    "message": "Beschreibung des Fehlers.",
    "status": 400
  }
}
```

Relevante HTTP-Statuscodes:

| Code | Bedeutung |
|---|---|
| `400` | Ungültige Anfrage (fehlende oder fehlerhafte Parameter) |
| `401` | Authentifizierung fehlgeschlagen |
| `403` | Fehlender Scope oder unzureichende Berechtigung |
| `404` | Ressource nicht gefunden |
| `429` | Rate Limit überschritten |
| `500` | Interner Serverfehler |

## Multi-Tenant-Modell

Die API arbeitet mit organisationsbasierter Mandantentrennung. Jeder API-Schlüssel ist genau einer Organisation zugeordnet. Anfragen liefern ausschließlich Daten der zugehörigen Organisation. Geräte, Bedrohungen und Alerts sind strikt isoliert.

## Integrationsmuster

### Pull-Integration (Polling)

Bei der Pull-Integration fragt Ihr System die API regelmäßig nach neuen Daten ab. Beachten Sie das Rate Limit.

```bash
curl -X GET "https://api.superheld.app/api/v1/threats?since=2026-03-14T09:00:00Z" \
  -H "Authorization: Bearer sh_live_abc123xyz456"
```

### Push-Integration (Webhooks)

Bei der Push-Integration sendet Superheld Ereignisse an Ihren HTTPS-Endpunkt. Konfiguration über Alert-Regeln mit `channel: "webhook"`:

```bash
curl -X POST https://api.superheld.app/api/v1/alerts \
  -H "Authorization: Bearer sh_live_abc123xyz456" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Kritische Bedrohungen",
    "severity_filter": ["high", "critical"],
    "channel": "webhook",
    "webhook_url": "https://mein-server.de/hooks/superheld"
  }'
```

Webhooks eignen sich für zeitkritische Sicherheitsereignisse. Polling eignet sich für regelmäßige Bestandsabgleiche. Beide Muster lassen sich kombinieren: Webhooks als primäre Quelle, Polling als Fallback.
