## Übersicht

Die Erkennungspipeline des Device Agent verarbeitet Signale in sechs aufeinander aufbauenden Stufen — von der Signalerfassung bis zur Alert-Ausgabe. Jede Stufe hat definierte Ein- und Ausgaben.

```mermaid
flowchart LR
    A["1 · Signal Collection"] --> B["2 · Local Detection"]
    B --> C["3 · Cloud Enrichment"]
    C --> D["4 · Policy Decision"]
    D --> E["5 · Event Generation"]
    E --> F["6 · Alert Exposure"]

    style A fill:#457b9d,color:#fff
    style B fill:#457b9d,color:#fff
    style C fill:#6c757d,color:#fff,stroke-dasharray:5 5
    style D fill:#e76f51,color:#fff
    style E fill:#2d6a4f,color:#fff
    style F fill:#2d6a4f,color:#fff
```

:::note
Stufe 3 (Cloud Enrichment) ist optional und gestrichelt dargestellt. Der Device Agent arbeitet vollständig lokal — Cloud Enrichment verbessert die Erkennung, ist aber nicht erforderlich.
:::

## 1 · Signal Collection — Signalerfassung

**Zweck:** Signale aus allen relevanten Quellen des Device sammeln und normalisieren.

| | Beschreibung |
|---|---|
| **Eingaben** | Betriebssystem-Events (Anrufstatus, App-Installationen, Berechtigungsänderungen), Netzwerkverkehr (DNS-Anfragen, Verbindungsaufbau), Benutzeraktionen (App-Starts, Eingaben in Sicherheitsdialoge) |
| **Ausgaben** | Normalisierte Signale mit Zeitstempel, Quelltyp und Kontextmetadaten |

Die Signalerfassung läuft als Hintergrundprozess auf dem Device. Signale werden in einem lokalen Ringpuffer zwischengespeichert, sodass auch bei hoher Systemlast keine Signale verloren gehen. Personenbezogene Inhalte werden bereits in dieser Stufe auf das für die Analyse notwendige Minimum reduziert.

---

## 2 · Local Detection — Lokale Erkennung

**Zweck:** Signale klassifizieren und Threat-Kandidaten identifizieren.

| | Beschreibung |
|---|---|
| **Eingaben** | Normalisierte Signale aus Stufe 1 |
| **Ausgaben** | Threat-Kandidaten mit Confidence Score (0.0–1.0), Bedrohungskategorie und erkannten Indikatoren |

**Confidence-Score-Schwellenwerte (Standard):**

| Confidence Score | Aktion |
|---|---|
| 0.0–0.3 | Allow |
| 0.3–0.7 | Warn |
| 0.7+ | Block |

> TODO: Standard-Schwellenwerte mit Produktteam bestätigen

Schwellenwerte sind über die Policy Engine konfigurierbar.

Die lokale Erkennung kombiniert regelbasierte Heuristiken mit kompakten ML-Modellen:

- **Heuristiken** — Bekannte Muster (z. B. Spoofing-Rufnummern, bekannte Betrugs-Domains) werden durch Regeln erkannt. Ziel-Inferenzzeit: < 10 ms.
- **ML-Modelle** — Für komplexere Fälle (Social-Engineering-Muster in Textnachrichten, verdächtiges App-Verhalten) kommen optimierte neuronale Netze zum Einsatz (quantisiert, < 50 MB). Ziel-Inferenzzeit: < 100 ms pro Signal. Gesamtlatenz Stufe 2: < 200 ms.

> TODO: Performance-Kennzahlen mit Engineering-Team bestätigen (Modellgröße < 50 MB, Heuristik < 10 ms, ML < 100 ms, Gesamtlatenz < 200 ms)

:::note
Alle Klassifikationen erfolgen lokal auf dem Device. Weder Nachrichteninhalte noch Anrufdaten verlassen das Device in dieser Stufe.
:::

## 3 · Cloud Enrichment — Cloud-Anreicherung (optional)

**Zweck:** Erkennungsqualität durch globale Threat Intelligence verbessern.

| | Beschreibung |
|---|---|
| **Eingaben** | (1) **SHA-256-Hashes** — für Threat-Intelligence-Lookups gegen die zentrale Threat Database. (2) **Anonymisierte Feature-Vektoren** — dimensionsreduzierte Repräsentationen bei Eskalation komplexer Fälle. TODO: Genaue Definition und Format der Feature-Vektoren dokumentieren. |
| **Ausgaben** | Angereicherte Threat-Daten — globale Verbreitungsinformationen, bekannte Kampagnenzuordnungen, aktuelle Indicator-of-Compromise-Daten |

Diese Stufe ist vollständig optional und kann vom Benutzer oder Administrator deaktiviert werden. Es werden zwei Arten von Daten an die Cloud gesendet:

1. **Kryptografische Hashes (SHA-256)** — z. B. von Telefonnummern oder App-Signaturen — werden gegen die Threat Database abgeglichen. Die Antwort enthält Risikobewertungen und Kontextinformationen (z. B. ob eine Rufnummer in bekannten Betrugskampagnen vorkommt).
2. **Anonymisierte Feature-Vektoren** — bei komplexen Fällen, die lokal nicht eindeutig klassifizierbar sind, werden Feature-Vektoren zur Cloud-Eskalation gesendet. Diese enthalten keine Klartextdaten.

## 4 · Policy Decision — Richtlinienentscheidung

**Zweck:** Konfigurierte Policies auf die Threat-Kandidaten anwenden.

| | Beschreibung |
|---|---|
| **Eingaben** | Threat-Kandidaten mit Confidence Scores + konfigurierte Policies (Schwellenwerte, Allowlists/Blocklists, organisationsspezifische Regeln) |
| **Ausgaben** | Aktionsentscheidung je Kandidat: **Allow**, **Warn** oder **Block** |

Die Policy Engine evaluiert jeden Threat-Kandidaten gegen die konfigurierte Policy-Hierarchie. In verwalteten Umgebungen (MDM) können Administratoren organisationsweite Mindeststandards definieren. Benutzer können innerhalb dieses Rahmens eigene Schwellenwerte anpassen — etwa die Empfindlichkeit der App-Berechtigungsprüfung erhöhen oder bestimmte Rufnummern freigeben.

:::note
Policy-Entscheidungen sind deterministisch und nachvollziehbar. Jede Entscheidung dokumentiert, welche Regel ausgelöst wurde und welcher Confidence Score zugrunde lag.
:::

## 5 · Event Generation — Ereigniserzeugung

**Zweck:** Unveränderlichen Audit-Trail für jede Entscheidung erstellen.

| | Beschreibung |
|---|---|
| **Eingaben** | Aktionsentscheidungen aus Stufe 4 |
| **Ausgaben** | Unveränderliche Events mit Zeitstempel, Entscheidung, angewandter Policy und Threat-Details |

Jede Entscheidung der Pipeline wird als unveränderlicher Event-Datensatz im lokalen Event Store gespeichert. Diese Events dienen als Audit-Trail für Compliance-Anforderungen und ermöglichen die nachträgliche Analyse von Sicherheitsvorfällen. Das Protokoll ist kryptografisch verkettet, sodass nachträgliche Manipulationen erkennbar sind.

---

## 6 · Alert Exposure — Alarmierung

**Zweck:** Benutzer und externe Systeme über Threats benachrichtigen.

| | Beschreibung |
|---|---|
| **Eingaben** | Events aus Stufe 5 |
| **Ausgaben** | Push-Benachrichtigungen (Alerts) an den Benutzer, Webhook-Payloads an konfigurierte Endpunkte, API-Antworten für SIEM-Integration |

Die Alert-Ausgabe stellt sicher, dass erkannte Threats die richtigen Empfänger erreichen. Endanwender erhalten kontextbezogene Alerts mit Handlungsempfehlungen. In Unternehmensumgebungen können Events per Webhook an SIEM-Systeme, Ticketing-Plattformen oder benutzerdefinierte Endpunkte weitergeleitet werden.

## Offline-Verhalten

Der Device Agent ist für den Offline-Betrieb konzipiert. Bei fehlender Netzwerkverbindung gilt:

| Stufe | Offline-Verfügbarkeit |
|---|---|
| 1 · Signal Collection | Vollständig lokal — keine Einschränkung |
| 2 · Local Detection | Vollständig lokal — keine Einschränkung |
| 3 · Cloud Enrichment | **Wird übersprungen** — kein Zugriff auf Threat Database und keine Feature-Vektor-Eskalation |
| 4 · Policy Decision | Vollständig lokal — keine Einschränkung |
| 5 · Event Generation | Vollständig lokal — Events werden im lokalen Event Store gespeichert |
| 6 · Alert Exposure | Lokale Alerts (Push-Benachrichtigungen) funktionieren. Webhooks und API-Zustellung werden zwischengespeichert und bei Wiederherstellung der Verbindung nachgeliefert. |
